## The Extension Model

Copilot CLI extensions are Node.js processes that join an active session via JSON-RPC over stdio. You create an `extension.mjs` file, place it in `.github/extensions/<name>/` (project-scoped) or the user's copilot config extensions directory (user-scoped), and the CLI discovers and launches it automatically.

The architecture:

```
┌─────────────────────┐          JSON-RPC / stdio           ┌──────────────────────┐
│   Copilot CLI        │ ◄──────────────────────────────────► │  Extension Process   │
│   (parent process)   │    tool calls, events, hooks        │  (forked child)      │
│                      │                                      │                      │
│  • Discovers exts    │                                      │  • Registers tools   │
│  • Forks processes   │                                      │  • Registers hooks   │
│  • Routes tool calls │                                      │  • Listens to events │
│  • Manages lifecycle │                                      │  • Uses SDK APIs     │
└─────────────────────┘                                      └──────────────────────┘
```

The entry point is `joinSession()` from `@github/copilot-sdk/extension`:

```js
import { joinSession } from "@github/copilot-sdk/extension";

const session = await joinSession({
  tools: [/* custom tools */],
  hooks: {/* lifecycle hooks */},
  onPermissionRequest: (request) => ({ permissionDecision: "allow" }),
});
```

That's the real API. The config accepts `tools`, `hooks`, `onPermissionRequest`, `onUserInputRequest`, and more. Extensions are reloaded on `/clear` and stopped on CLI exit (SIGTERM, then SIGKILL after 5s).

**Important:** Only `.mjs` files are supported. The file must be named `extension.mjs`. The `@github/copilot-sdk` import is resolved automatically — you don't install it.

### Extension Discovery and Management

Extensions live in two places:

- **Project extensions:** `.github/extensions/<name>/extension.mjs` — scoped to the repo
- **User extensions:** copilot config extensions directory — available everywhere

Project extensions shadow user extensions on name collision. Only immediate subdirectories are checked.

The CLI provides a real `/extensions` slash command for managing extensions in-session, and an `extensions_manage` tool that agents (and extensions themselves) can call:

```
extensions_manage({ operation: "list" })           // List all loaded extensions
extensions_manage({ operation: "inspect", name: "my-ext" })  // Extension details
extensions_manage({ operation: "scaffold", name: "my-ext" }) // Create skeleton
extensions_manage({ operation: "guide" })           // Get authoring guidance
```

Extensions support three modes:
- **`load_and_augment`** — loaded and active (tools and hooks registered)
- **`load_only`** — loaded but not contributing tools/hooks
- **`disabled`** — not loaded at all

You can reload all extensions mid-session with `extensions_reload({})` — no context loss, no restart. Only the extension processes restart. This makes iterative development viable: edit your `extension.mjs`, reload, test, repeat — all in the same conversation.

## Custom Tools with `skipPermission`

Tools are the primary way extensions add capabilities. Each tool gets a name, description, optional parameter schema, and a handler:

```js
import { joinSession, defineTool } from "@github/copilot-sdk/extension";
import { z } from "zod";

const gitStatusTool = defineTool("get_git_status", {
  description: "Get current git status",
  parameters: z.object({
    short: z.boolean().optional().describe("Use short format"),
  }),
  skipPermission: true,
  handler: async ({ short }) => {
    const { execSync } = await import("child_process");
    const flag = short ? "--porcelain" : "";
    return execSync(`git status ${flag}`).toString();
  },
});

await joinSession({
  tools: [gitStatusTool],
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

The `skipPermission: true` flag tells the CLI to execute the tool without prompting the user. Use it for read-only, side-effect-free operations. For anything that writes files or runs arbitrary commands, leave it off.

**Important constraint:** Tool names must be globally unique across all loaded extensions. If two extensions register the same tool name, the second extension fails to initialize. Plan your naming accordingly.

### Tool Handler Context

Handlers receive a second argument with invocation metadata:

```js
handler: async (args, invocation) => {
  // invocation.sessionId  — current session ID
  // invocation.toolCallId — unique ID for this tool call
  // invocation.toolName   — name of the tool being called
  return "done";
}
```

### Tool Results

Handlers can return a simple string (treated as success) or a structured result:

```js
handler: async (args) => {
  // Simple: just return a string
  return "Operation completed successfully";

  // Structured: more control over what the agent sees
  return {
    textResultForLlm: "Found 3 security issues",
    resultType: "success",  // "success" | "failure" | "rejected" | "denied"
  };
}
```

## Lifecycle Hooks

The `hooks` object gives you six interception points in the session lifecycle:

### `onPreToolUse` — The Security Guardrail

The most powerful hook. Intercept every tool call before it executes:

```js
hooks: {
  onPreToolUse: async (input) => {
    if (input.toolName === "bash") {
      const cmd = String(input.toolArgs?.command || "");
      if (/rm\s+-rf/i.test(cmd)) {
        return {
          permissionDecision: "deny",
          permissionDecisionReason: "Destructive commands are not allowed.",
        };
      }
    }
    // Can also modify args before execution:
    // return { modifiedArgs: { ...input.toolArgs, safe: true } };
    // Or inject context:
    // return { additionalContext: "Remember to handle errors." };
  },
}
```

### `onPostToolUse` — React to Results

Called after tool execution with the full result:

```js
hooks: {
  onPostToolUse: async (input) => {
    if (input.toolName === "edit") {
      const filePath = input.toolArgs?.path;
      if (filePath?.endsWith(".ts")) {
        const { exec } = await import("child_process");
        const result = await new Promise((resolve) => {
          exec(`npx eslint "${filePath}"`, (err, stdout) => {
            resolve(err ? stdout : "No lint errors.");
          });
        });
        return { additionalContext: `Lint result: ${result}` };
      }
    }
  },
}
```

### `onUserPromptSubmitted` — Enrich Every Prompt

Automatically inject context or modify the user's message:

```js
hooks: {
  onUserPromptSubmitted: async (input) => {
    return {
      additionalContext: "Always follow the team's TypeScript style guide. Use 4-space indentation.",
      // Or rewrite the prompt entirely:
      // modifiedPrompt: input.prompt.toUpperCase(),
    };
  },
}
```

### `onSessionStart` and `onSessionEnd`

```js
hooks: {
  onSessionStart: async (input) => {
    // input.source: "startup" | "resume" | "new"
    await session.log("Security extension active");
    return { additionalContext: "Security policies loaded." };
  },
  onSessionEnd: async (input) => {
    // input.reason: "complete" | "error" | "abort" | "timeout" | "user_exit"
    return { sessionSummary: "Session completed with no policy violations." };
  },
}
```

### `onErrorOccurred` — Retry Logic

```js
hooks: {
  onErrorOccurred: async (input) => {
    // input.errorContext: "model_call" | "tool_execution" | "system" | "user_input"
    if (input.recoverable && input.errorContext === "model_call") {
      return { errorHandling: "retry", retryCount: 2 };
    }
    return {
      errorHandling: "abort",
      userNotification: `An error occurred: ${input.error}`,
    };
  },
}
```

## Elicitation: Structured User Input

This is a real feature — the previous article just got the API shape wrong. Elicitation is **event-based**, not a method call. The agent emits an `elicitation.requested` event carrying a JSON Schema that describes the form fields:

```js
session.on("elicitation.requested", (event) => {
  const { requestId, message, mode, requestedSchema } = event.data;
  // mode: currently only "form"
  // requestedSchema: JSON Schema with type, properties, required
  console.error(`Agent asks: ${message}`);
  console.error(`Fields:`, JSON.stringify(requestedSchema.properties, null, 2));

  // Client/extension collects user input, then responds
});

session.on("elicitation.completed", (event) => {
  // event.data.requestId — dismiss any UI for this request
});
```

The canonical use case: the agent needs a structured decision before proceeding. Instead of hoping the user types the right thing in natural language, elicitation requests specific fields — environment selectors, confirmation booleans, change descriptions. The schema enforces structure; the event flow keeps it non-blocking.

This is particularly valuable for governance workflows. When the agent tries to deploy or modify production configs, you want explicit, structured confirmation — not an inferred intent from a chat message.

## MCP Server Integration

Extensions can bring external tools into the session through MCP (Model Context Protocol) servers. This is configured at the session level via `mcpServers`:

```js
await joinSession({
  mcpServers: {
    "my-db-tools": {
      type: "local",
      command: "node",
      args: ["./mcp-servers/db-tools.js"],
      env: { DATABASE_URL: process.env.DATABASE_URL },
    },
  },
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

MCP is the real multi-language extensibility story. An MCP server can be written in any language — Python, Go, Rust, whatever implements the protocol. The extension handles the SDK communication in Node.js; MCP servers handle the business logic in whatever makes sense.

## Custom Agents

The `customAgents` config in `SessionConfig` defines specialized agents accessible via `@agentname`:

```js
customAgents: [
  {
    name: "security-reviewer",
    displayName: "Security Reviewer",
    description: "Reviews code for security vulnerabilities",
    prompt: `You are a security-focused code reviewer. Analyze code for:
      - Hardcoded credentials and secrets
      - SQL injection and XSS vulnerabilities
      - Insecure dependency usage
      Report findings with severity levels.`,
    tools: ["read_file", "search_files", "get_git_diff"],
    mcpServers: { /* agent-specific MCP servers */ },
  },
]
```

Each agent gets its own prompt, tool subset, and MCP servers. Users switch between agents during the session with `@agentname`. This is functionally equivalent to custom commands — you type `@security-reviewer` and get specialized behavior scoped to that agent's capabilities.

## The Session Object

After `joinSession()` resolves, you get a `CopilotSession` with these capabilities:

### Sending Messages

```js
// Fire and forget
await session.send({ prompt: "Analyze the test results." });

// With file attachments
await session.send({
  prompt: "Review this file",
  attachments: [{ type: "file", path: "./src/index.ts" }],
});

// Send and wait for the agent to finish
const response = await session.sendAndWait(
  { prompt: "What is 2+2?" },
  60000
);
console.error(response?.data.content);
```

### Logging to the Timeline

```js
await session.log("Extension ready");
await session.log("Rate limit approaching", { level: "warning" });
await session.log("Connection failed", { level: "error" });
await session.log("Debug info", { ephemeral: true });
```

**Don't use `console.log()`** — stdout is reserved for JSON-RPC. Use `session.log()` for user-visible messages, `console.error()` for debug output.

### Event Subscription

```js
const unsub = session.on("assistant.message", (event) => {
  console.error("Agent:", event.data.content);
});

// Key event types:
// assistant.message — agent's final response
// tool.execution_start / tool.execution_complete — tool lifecycle
// user.message — user input
// session.idle — agent finished processing
// session.error — something went wrong
// permission.requested — permission check needed
// elicitation.requested / elicitation.completed — structured input
// session.shutdown — session ending
// command.queued — slash command dispatch

unsub(); // stop listening
```

### Model Switching

```js
await session.setModel("gpt-4.1");
```

## The Permission System

The `onPermissionRequest` handler receives requests with a `kind` field:

```js
await joinSession({
  onPermissionRequest: (request) => {
    switch (request.kind) {
      case "shell":
        return { permissionDecision: "deny" };
      case "write":
        return { permissionDecision: "ask" };
      default:
        return { permissionDecision: "allow" };
    }
  },
});
```

The `onUserInputRequest` handler handles agent questions (the `ask_user` tool):

```js
await joinSession({
  onUserInputRequest: async (request) => {
    // request.question, request.choices
    return { answer: "yes", wasFreeform: false };
  },
});
```

## What's Coming: Custom Slash Commands

I've confirmed with the CLI team that **custom slash command registration for extensions is planned**. The mechanism already exists internally — Skills can register slash commands today via `createSkillSlashCommand`, and the internal architecture supports `handlePendingCommand` / `respondToCommandExecution` in the RPC layer.

What's missing is exposing this as a `slashCommands` property in `joinSession()` for extensions. When it ships, it'll probably look something like:

```js
// FUTURE — not available in v1.0.11
await joinSession({
  slashCommands: [
    {
      name: "security-check",
      description: "Run a security audit",
      action: async (session, params) => {
        await session.send({
          prompt: "Run a security review of the current changes.",
        });
      },
    },
  ],
  tools: [],
});
```

In the meantime, you have three paths to command-like behavior:

1. **Custom Agents** — `@agentname` gives you specialized behavior with a specific prompt and tool set
2. **Skills** — the `/skills` system already registers actual slash commands from skill directories
3. **Tool-based workflows** — define tools that the agent calls on your behalf

Custom agents are the closest equivalent today. Each one is effectively a named entry point with its own prompt, tools, and MCP servers.

## Scaffolding a New Extension

The fastest path from zero to working extension:

1. In a Copilot CLI session, have the agent call `extensions_manage({ operation: "scaffold", name: "my-ext" })` — creates `.github/extensions/my-ext/extension.mjs` with a skeleton.

2. For user-scoped extensions, add `location: "user"`.

3. Edit the generated file. Add your tools and hooks.

4. Reload with `extensions_reload({})` — loads immediately, mid-session.

5. Verify with `extensions_manage({ operation: "list" })` and `extensions_manage({ operation: "inspect", name: "my-ext" })`.

The whole cycle — scaffold, edit, reload, test — happens in one session without losing context.

## What to Build

Given what the SDK *actually* offers today:

**Security guardrails via `onPreToolUse`**: Intercept every tool call. Block dangerous shell commands. Deny writes to protected paths. Log everything. This is the highest-value extension pattern — it directly addresses the "AI agent ran amok" risk that keeps security teams up at night.

**Domain-specific tools via MCP**: Stand up an MCP server in whatever language your team uses. Expose internal APIs, database queries, monitoring dashboards as tools the agent can call. The extension registers the MCP server; the agent gets access to your entire internal toolchain.

**Custom agents for specialized workflows**: A `@security-reviewer` agent, a `@db-admin` agent, a `@docs-writer` agent — each with a focused prompt and restricted tool set. Users switch between them during the session.

**Prompt enrichment via `onUserPromptSubmitted`**: Automatically inject team context, coding standards, or relevant documentation into every prompt. The user types their question; your hook adds what they'd otherwise forget.

**Governance workflows via elicitation events**: When agents need human decisions — deployment targets, destructive operations, environment selection — listen for `elicitation.requested` events and build structured approval flows instead of relying on natural language.

## The Bottom Line

The Copilot CLI extension SDK is genuinely powerful. `joinSession()` with custom tools, six lifecycle hooks, MCP integration, custom agents, event-based elicitation, permission control, and real in-session extension management covers serious ground.

Custom slash commands for extensions are confirmed coming. Until they ship, custom agents (`@agentname`) and the skills system cover the same use case from different angles. The foundation is here. Build on what's real — there's plenty of it.
