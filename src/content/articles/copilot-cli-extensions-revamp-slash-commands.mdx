## A Correction

The previous version of this article contained fabricated API surfaces. I'm not going to pretend it didn't happen — the original described features like `slashCommands` in `joinSession()`, `session.ui.elicitation()` dialogs, a `/extensions` CLI command, and a multi-language Python SDK with `join_session()`. **None of those exist.** They were hallucinated during content generation and published without proper verification against the actual SDK source.

This rewrite is based entirely on the real `@github/copilot-sdk` TypeScript type definitions. Every code example below comes from or is derived from the actual exported types. If it's not in the types, it's not in this article.

## What the Extension SDK Actually Is

The Copilot CLI extension SDK (`@github/copilot-sdk`) lets you build extensions that join an active Copilot CLI session as a child process communicating over JSON-RPC (stdio or TCP). The entry point is `joinSession()` from `@github/copilot-sdk/extension`:

```typescript
import { joinSession } from "@github/copilot-sdk/extension";

const session = await joinSession({
  tools: [/* your custom tools */],
  hooks: {/* lifecycle hooks */},
  customAgents: [/* agent definitions */],
  mcpServers: {/* MCP server configs */},
  onPermissionRequest: (request) => ({ permissionDecision: "allow" }),
});
```

That's the real API. The config accepts `tools`, `hooks`, `customAgents`, `mcpServers`, `systemMessage`, `availableTools`, `excludedTools`, `skillDirectories`, `infiniteSessions`, and a few other fields. There is no `slashCommands` property. There is no `ui` property on the session object.

## Custom Tools with `skipPermission`

Tools are the primary way extensions add capabilities. Each tool gets a name, description, optional parameter schema (raw JSON Schema or Zod), and a handler function:

```typescript
import { joinSession, defineTool } from "@github/copilot-sdk/extension";
import { z } from "zod";

const gitStatusTool = defineTool("get_git_status", {
  description: "Get current git status",
  parameters: z.object({
    short: z.boolean().optional().describe("Use short format"),
  }),
  skipPermission: true, // No permission prompt for read-only ops
  handler: async ({ short }) => {
    const flag = short ? "--porcelain" : "";
    return execSync(`git status ${flag}`).toString();
  },
});

await joinSession({
  tools: [gitStatusTool],
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

The `skipPermission: true` flag is real and useful — it tells the CLI to execute the tool without prompting the user. Use it for read-only, side-effect-free operations. For anything that writes files or runs arbitrary commands, leave it off and let the permission system do its job.

You can also set `overridesBuiltInTool: true` if you intentionally want to replace a built-in tool with your own implementation.

## Lifecycle Hooks

The `hooks` object in `SessionConfig` gives you six interception points:

```typescript
await joinSession({
  hooks: {
    onPreToolUse: async (input) => {
      // Called before any tool executes
      // Can modify args, override permission, add context
      if (input.toolName === "shell_exec" && containsDangerousCommand(input.toolArgs)) {
        return {
          permissionDecision: "deny",
          permissionDecisionReason: "Command blocked by security policy",
        };
      }
    },

    onPostToolUse: async (input) => {
      // Called after tool execution with the result
      // Can modify result, add context, suppress output
      console.log(`Tool ${input.toolName} completed: ${input.toolResult.resultType}`);
    },

    onUserPromptSubmitted: async (input) => {
      // Called when user sends a message
      // Can modify the prompt or inject additional context
      return {
        additionalContext: "Remember: follow the team's coding standards.",
      };
    },

    onSessionStart: async (input) => {
      // source: "startup" | "resume" | "new"
      console.log(`Session started via ${input.source}`);
    },

    onSessionEnd: async (input) => {
      // reason: "complete" | "error" | "abort" | "timeout" | "user_exit"
      if (input.reason === "error") {
        await logError(input.error);
      }
    },

    onErrorOccurred: async (input) => {
      // errorContext: "model_call" | "tool_execution" | "system" | "user_input"
      if (input.recoverable) {
        return { errorHandling: "retry", retryCount: 2 };
      }
      return { errorHandling: "abort" };
    },
  },
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

The `onPreToolUse` hook is the most powerful — it can allow, deny, or defer to the user (`"ask"`) on a per-invocation basis, modify tool arguments before execution, and inject additional context. This is where you build guardrails.

## MCP Server Integration

Extensions can register MCP (Model Context Protocol) servers — local processes or remote endpoints:

```typescript
await joinSession({
  mcpServers: {
    // Local process over stdio
    "my-db-tools": {
      type: "local",
      command: "node",
      args: ["./mcp-servers/db-tools.js"],
      env: { DATABASE_URL: process.env.DATABASE_URL },
      tools: ["*"], // expose all tools from this server
    },

    // Remote HTTP server
    "team-api": {
      type: "http",
      url: "https://internal.example.com/mcp",
      headers: { Authorization: `Bearer ${process.env.API_TOKEN}` },
      tools: ["search_docs", "get_runbook"], // expose only specific tools
      timeout: 30000,
    },

    // Remote SSE server
    "monitoring": {
      type: "sse",
      url: "https://monitoring.example.com/mcp-sse",
      tools: ["get_metrics", "get_alerts"],
    },
  },
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

MCP is the real extensibility story for bringing external tools into Copilot CLI. It's an open standard — any MCP-compatible server works, regardless of what language it's written in. This is the "multi-language" story, not a mythical Python SDK.

## Custom Agents

The `customAgents` config lets you define specialized agents with their own prompts, tool restrictions, and MCP servers:

```typescript
await joinSession({
  customAgents: [
    {
      name: "security-reviewer",
      displayName: "Security Reviewer",
      description: "Reviews code for security vulnerabilities",
      prompt: `You are a security-focused code reviewer. Analyze code for:
        - Hardcoded credentials and secrets
        - SQL injection and XSS vulnerabilities
        - Insecure dependency usage
        - Missing input validation
        Report findings with severity levels.`,
      tools: ["read_file", "search_files", "get_git_diff"],
      infer: true,
    },
    {
      name: "db-admin",
      displayName: "Database Admin",
      description: "Manages database operations",
      prompt: "You are a database administration specialist...",
      mcpServers: {
        "db-tools": {
          type: "local",
          command: "node",
          args: ["./db-mcp.js"],
          tools: ["*"],
        },
      },
    },
  ],
  agent: "security-reviewer", // activate this agent on session start
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

The `agent` field in `SessionConfig` selects which custom agent to activate at startup — equivalent to calling `session.rpc.agent.select()` after creation. Users can switch agents during the session with `/agent`.

## Session RPC: What the Session Object Exposes

Once you have a `CopilotSession` from `joinSession()`, you get access to the RPC interface for session management. Based on the SDK types, this includes:

- **Model switching** — change the active model mid-session
- **Plan management** — create and manage implementation plans
- **Workspace operations** — file and directory operations scoped to the session
- **Fleet mode** — parallel subagent execution
- **Agent selection** — switch between custom agents
- **Compaction** — summarize conversation to manage context window
- **Shell execution** — run commands within the session's permission scope

The `InfiniteSessionConfig` is worth calling out — it enables automatic context compaction when you hit thresholds:

```typescript
await joinSession({
  infiniteSessions: {
    enabled: true,
    backgroundCompactionThreshold: 0.80, // start compacting at 80% context usage
    bufferExhaustionThreshold: 0.95,      // block until compaction at 95%
  },
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

## The Permission System

Permissions in Copilot CLI are tool-level and path-level, not extension-level. The `onPermissionRequest` handler receives requests with a `kind` field:

- `"shell"` — shell command execution
- `"write"` — file write operations
- `"read"` — file read operations
- `"mcp"` — MCP tool invocations
- `"url"` — URL access
- `"custom-tool"` — your extension's custom tools

You can use the built-in `approveAll` helper for development, or implement granular logic:

```typescript
import { approveAll } from "@github/copilot-sdk";

// For development:
await joinSession({ onPermissionRequest: approveAll });

// For production:
await joinSession({
  onPermissionRequest: (request) => {
    if (request.kind === "shell") {
      return { permissionDecision: "deny" };
    }
    if (request.kind === "write") {
      return { permissionDecision: "ask" }; // let user decide
    }
    return { permissionDecision: "allow" };
  },
});
```

## System Message Control

You can append to or fully replace the system message:

```typescript
// Append mode (default) — adds to the SDK's built-in system message
await joinSession({
  systemMessage: {
    mode: "append",
    content: "Always follow the team's TypeScript style guide.",
  },
  onPermissionRequest: approveAll,
});

// Replace mode — full control, removes SDK guardrails
await joinSession({
  systemMessage: {
    mode: "replace",
    content: "You are a specialized security auditor. Only analyze code for vulnerabilities.",
  },
  onPermissionRequest: approveAll,
});
```

Replace mode is powerful but risky — it removes all SDK-managed safety instructions. Use it only when you know what you're doing.

## BYOK: Bring Your Own Key

The `provider` config lets you point the session at your own API endpoint:

```typescript
await joinSession({
  provider: {
    type: "openai",      // or "azure" or "anthropic"
    baseUrl: "http://localhost:11434/v1", // Ollama, for example
    apiKey: "optional-for-local",
  },
  onPermissionRequest: approveAll,
});
```

This works with any OpenAI-compatible API. Azure gets its own config with `apiVersion`. Anthropic is supported as a provider type. You can also set `wireApi: "responses"` for the newer Responses API format.

## What to Actually Build

Given what the SDK *really* offers, here's what's worth building:

**Security guardrails via `onPreToolUse`**: Intercept every tool call. Block dangerous shell commands. Deny writes to protected paths. Log everything. This is the highest-value extension pattern — it works today, it's straightforward, and it directly addresses the "AI agent ran amok" risk.

**Domain-specific tools via MCP**: Stand up an MCP server in whatever language your team uses. Expose your internal APIs, database queries, monitoring dashboards as tools the agent can call. MCP is the real multi-language story.

**Custom agents for specialized workflows**: Define a `security-reviewer` agent, a `db-admin` agent, a `docs-writer` agent. Each gets a focused prompt and restricted tool set. Users switch between them with `/agent` based on what they're doing.

**Prompt enrichment via `onUserPromptSubmitted`**: Automatically inject team context, coding standards, or relevant documentation into every prompt. The user types their question; your hook adds the context they'd otherwise forget to include.

## The Real Extensibility Story

The Copilot CLI extension SDK is genuinely powerful — `joinSession()` with custom tools, six lifecycle hooks, MCP integration, custom agents, permission control, infinite sessions, and BYOK support covers a lot of ground. You don't need fabricated features to make the case.

What the SDK *doesn't* have (yet) is a CLI-level extension management command, custom slash commands registered by extensions, or structured UI dialogs. Those might come — but they're not here today, and the previous version of this article was wrong to present them as if they were.

Build on what's real. The hooks and tools system is already enough to create meaningful governance, security, and productivity extensions for your team.
