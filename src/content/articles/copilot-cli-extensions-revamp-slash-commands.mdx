## The Extension Model

Copilot CLI extensions are Node.js processes that join an active session via JSON-RPC (stdio). You create an `extension.mjs` file, place it in `.github/extensions/<name>/` (project-scoped) or `~/.copilot/extensions/<name>/` (user-scoped), and the CLI discovers and launches it automatically.

The entry point is `joinSession()` from `@github/copilot-sdk/extension`:

```js
import { joinSession } from "@github/copilot-sdk/extension";

const session = await joinSession({
  tools: [/* custom tools */],
  hooks: {/* lifecycle hooks */},
  customAgents: [/* agent definitions */],
  mcpServers: {/* MCP server configs */},
  onPermissionRequest: (request) => ({ permissionDecision: "allow" }),
});
```

That's the real API. The config accepts `tools`, `hooks`, `customAgents`, `mcpServers`, `systemMessage`, `availableTools`, `excludedTools`, `skillDirectories`, `infiniteSessions`, and more. There is no `slashCommands` property.

### Extension Discovery and Management

Extensions live in two places:

- **Project extensions:** `.github/extensions/<name>/extension.mjs` — scoped to the repo
- **User extensions:** `~/.copilot/extensions/<name>/extension.mjs` — available everywhere

Project extensions shadow user extensions on name collision. Only `.mjs` files are supported — no TypeScript, no CommonJS.

The CLI provides a real `/extensions` slash command for managing extensions in-session, and an `extensions_manage` tool that agents (and extensions themselves) can call:

```
extensions_manage({ operation: "list" })           // List all loaded extensions
extensions_manage({ operation: "inspect", name: "my-ext" })  // Extension details
extensions_manage({ operation: "scaffold", name: "my-ext" }) // Create skeleton
extensions_manage({ operation: "guide" })           // Get authoring guidance
```

Extensions support three modes:
- **`load_and_augment`** — loaded and active (tools and hooks registered)
- **`load_only`** — loaded but not contributing tools/hooks
- **`disabled`** — not loaded at all

You can reload all extensions mid-session with `extensions_reload({})` — no context loss, no restart. Only the extension processes restart. This makes iterative development viable: edit your `extension.mjs`, reload, test, repeat — all in the same conversation.

## Custom Tools with `skipPermission`

Tools are the primary way extensions add capabilities. Each tool gets a name, description, optional parameter schema (raw JSON Schema or Zod), and a handler:

```js
import { joinSession, defineTool } from "@github/copilot-sdk/extension";
import { z } from "zod";

const gitStatusTool = defineTool("get_git_status", {
  description: "Get current git status",
  parameters: z.object({
    short: z.boolean().optional().describe("Use short format"),
  }),
  skipPermission: true, // No permission prompt for read-only ops
  handler: async ({ short }) => {
    const { execSync } = await import("child_process");
    const flag = short ? "--porcelain" : "";
    return execSync(`git status ${flag}`).toString();
  },
});

await joinSession({
  tools: [gitStatusTool],
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

The `skipPermission: true` flag tells the CLI to execute the tool without prompting the user. Use it for read-only, side-effect-free operations. For anything that writes files or runs arbitrary commands, leave it off.

You can also set `overridesBuiltInTool: true` if you intentionally want to replace a built-in tool with your own implementation — without it, a name collision with a built-in tool is an error.

**Important constraint:** Tool names must be globally unique across all loaded extensions. If two extensions register the same tool name, the second extension fails to initialize. Plan your naming accordingly.

### Tool Results

Handlers can return a simple string (treated as success) or a structured `ToolResultObject`:

```js
handler: async (args) => {
  // Simple: just return a string
  return "Operation completed successfully";

  // Structured: more control over what the agent sees
  return {
    textResultForLlm: "Found 3 security issues",
    resultType: "success",  // "success" | "failure" | "rejected" | "denied"
    sessionLog: "Scanned 42 files",
    binaryResultsForLlm: [{
      data: base64EncodedImage,
      mimeType: "image/png",
      type: "image",
      description: "Security scan results chart",
    }],
  };
}
```

## Lifecycle Hooks

The `hooks` object gives you six interception points in the session lifecycle:

```js
await joinSession({
  hooks: {
    onPreToolUse: async (input) => {
      // Called before any tool executes
      // Can modify args, override permission, add context
      if (input.toolName === "shell_exec" && isDangerous(input.toolArgs)) {
        return {
          permissionDecision: "deny",
          permissionDecisionReason: "Blocked by security policy",
        };
      }
      // Can also modify args before execution:
      // return { modifiedArgs: { ...input.toolArgs, safe: true } };
    },

    onPostToolUse: async (input) => {
      // Called after tool execution with the full result
      // input.toolResult has: textResultForLlm, resultType, error
      if (input.toolResult.resultType === "failure") {
        await logFailure(input.toolName, input.toolResult.error);
      }
      // Can modify the result before the agent sees it:
      // return { modifiedResult: { ...input.toolResult, textResultForLlm: "sanitized" } };
    },

    onUserPromptSubmitted: async (input) => {
      // Called when the user sends a message
      // input.prompt contains the raw user input
      return {
        additionalContext: "Remember: follow the team's coding standards.",
        // Or modify the prompt entirely:
        // modifiedPrompt: `${input.prompt}\n\nContext: we use ESLint strict mode`,
      };
    },

    onSessionStart: async (input) => {
      // input.source: "startup" | "resume" | "new"
      // input.initialPrompt: the first prompt (if any)
      await session.log("Security extension active");
      return { additionalContext: "Security policies loaded." };
    },

    onSessionEnd: async (input) => {
      // input.reason: "complete" | "error" | "abort" | "timeout" | "user_exit"
      if (input.reason === "error") {
        await reportError(input.error);
      }
      return { sessionSummary: "Session completed with 3 tool invocations" };
    },

    onErrorOccurred: async (input) => {
      // input.errorContext: "model_call" | "tool_execution" | "system" | "user_input"
      // input.recoverable: boolean
      if (input.recoverable) {
        return { errorHandling: "retry", retryCount: 2 };
      }
      return {
        errorHandling: "abort",
        userNotification: `Fatal error: ${input.error}`,
      };
    },
  },
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

The `onPreToolUse` hook is the most powerful — it can allow, deny, or defer to the user (`"ask"`) on a per-invocation basis, modify arguments before execution, and inject additional context. This is where you build security guardrails.

## Elicitation: Structured User Input (Event-Based)

This is a real feature, but the previous article got the API wrong. Elicitation is **event-based**, not a method call. The agent triggers an `elicitation.requested` event, the client/extension listens for it and presents a form to the user, then responds via `respondToElicitation()`.

The event carries a JSON Schema describing the form fields:

```js
session.on("elicitation.requested", async (event) => {
  const { requestId, message, requestedSchema } = event.data;
  // requestedSchema is a JSON Schema with:
  //   type: "object"
  //   properties: { fieldName: { type, description, ... }, ... }
  //   required: ["fieldName", ...]

  console.error(`Agent asks: ${message}`);
  console.error(`Fields:`, JSON.stringify(requestedSchema.properties, null, 2));

  // Collect user input somehow, then respond:
  await session.respondToElicitation(requestId, {
    // key-value pairs matching the requestedSchema
    environment: "production",
    confirmDeploy: true,
  });
});

// When the agent is done with the form:
session.on("elicitation.completed", (event) => {
  // event.data.requestId — dismiss any UI for this request
});
```

The canonical use case: the agent needs a structured decision before proceeding. Instead of hoping the user types the right thing in natural language, the agent requests specific fields — environment selectors, confirmation booleans, change descriptions. The schema enforces structure; the event flow keeps it non-blocking.

This is particularly valuable for governance workflows: when the agent tries to deploy or modify production configs, you want explicit, structured confirmation — not an inferred intent from a chat message.

## MCP Server Integration

Extensions can register MCP (Model Context Protocol) servers — local processes or remote endpoints:

```js
await joinSession({
  mcpServers: {
    // Local process over stdio
    "my-db-tools": {
      type: "local",
      command: "node",
      args: ["./mcp-servers/db-tools.js"],
      env: { DATABASE_URL: process.env.DATABASE_URL },
      tools: ["*"], // expose all tools from this server
    },

    // Remote HTTP server
    "team-api": {
      type: "http",
      url: "https://internal.example.com/mcp",
      headers: { Authorization: `Bearer ${process.env.API_TOKEN}` },
      tools: ["search_docs", "get_runbook"], // only specific tools
      timeout: 30000,
    },

    // Remote SSE server
    "monitoring": {
      type: "sse",
      url: "https://monitoring.example.com/mcp-sse",
      tools: ["get_metrics", "get_alerts"],
    },
  },
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

MCP is the real multi-language extensibility story. An MCP server can be written in any language — Python, Go, Rust, whatever implements the protocol. The extension handles the SDK communication in Node.js; MCP servers handle the business logic in whatever makes sense.

## Custom Agents

The `customAgents` config lets you define specialized agents with their own prompts, tool restrictions, and MCP servers:

```js
await joinSession({
  customAgents: [
    {
      name: "security-reviewer",
      displayName: "Security Reviewer",
      description: "Reviews code for security vulnerabilities",
      prompt: `You are a security-focused code reviewer. Analyze code for:
        - Hardcoded credentials and secrets
        - SQL injection and XSS vulnerabilities
        - Insecure dependency usage
        - Missing input validation
        Report findings with severity levels.`,
      tools: ["read_file", "search_files", "get_git_diff"],
      infer: true, // available for model inference
    },
    {
      name: "db-admin",
      displayName: "Database Admin",
      description: "Manages database operations",
      prompt: "You are a database administration specialist...",
      mcpServers: {
        "db-tools": {
          type: "local",
          command: "node",
          args: ["./db-mcp.js"],
          tools: ["*"],
        },
      },
    },
  ],
  agent: "security-reviewer", // activate on session start
  onPermissionRequest: () => ({ permissionDecision: "allow" }),
});
```

The `agent` field selects which custom agent to activate at startup — equivalent to calling `session.rpc.agent.select()` after creation. Users switch between agents during the session with `/agent`.

## The Session Object

After `joinSession()` resolves, you get a `CopilotSession` with these capabilities:

### Sending Messages

```js
// Fire and forget — returns message ID
await session.send({ prompt: "Analyze the test results." });

// Send with file attachments
await session.send({
  prompt: "Review this file",
  attachments: [{ type: "file", path: "./src/index.ts" }],
});

// Send and wait for the agent to finish (blocks until session.idle)
const response = await session.sendAndWait(
  { prompt: "What is 2+2?" },
  60000 // timeout in ms
);
console.error(response?.data.content); // "4"
```

### Logging to the Timeline

```js
await session.log("Extension ready");
await session.log("Rate limit approaching", { level: "warning" });
await session.log("Connection failed", { level: "error" });
await session.log("Debug info", { ephemeral: true }); // transient, not persisted
```

**Important:** Don't use `console.log()` — stdout is reserved for JSON-RPC. Use `session.log()` to surface messages, or `console.error()` for debug output.

### Event Subscription

```js
const unsub = session.on("assistant.message", (event) => {
  console.error("Agent:", event.data.content);
});

// Key event types:
// assistant.message — agent's response
// tool.execution_start / tool.execution_complete — tool lifecycle
// user.message — user input
// session.idle — agent finished processing
// session.error — something went wrong
// permission.requested — permission check needed
// elicitation.requested / elicitation.completed — structured user input
// session.shutdown — session ending

// Unsubscribe when done
unsub();
```

### Model Switching and RPC

```js
await session.setModel("gpt-4.1");

// Low-level RPC access for advanced operations:
// session.rpc — typed access to model, plan, workspace, agent, compaction APIs
```

## The Permission System

Permissions are tool-level and path-level. The `onPermissionRequest` handler receives requests with a `kind` field:

```js
import { approveAll } from "@github/copilot-sdk";

// For development — approve everything:
await joinSession({ onPermissionRequest: approveAll });

// For production — granular control:
await joinSession({
  onPermissionRequest: (request) => {
    // request.kind: "shell" | "write" | "read" | "mcp" | "url" | "custom-tool"
    switch (request.kind) {
      case "shell":
        return { permissionDecision: "deny" }; // no shell access
      case "write":
        return { permissionDecision: "ask" };  // let user decide
      default:
        return { permissionDecision: "allow" };
    }
  },
});
```

## System Message Control

You get two modes — append to or fully replace the system message:

```js
// Append mode (default) — adds to the SDK's built-in system message
await joinSession({
  systemMessage: {
    mode: "append",
    content: "Always follow the team's TypeScript style guide.",
  },
  onPermissionRequest: approveAll,
});

// Replace mode — full control, removes ALL SDK guardrails
await joinSession({
  systemMessage: {
    mode: "replace",
    content: "You are a specialized security auditor. Only analyze code for vulnerabilities.",
  },
  onPermissionRequest: approveAll,
});
```

Replace mode is powerful but risky — it removes all SDK-managed safety instructions. Know what you're doing.

## Infinite Sessions

The `InfiniteSessionConfig` enables automatic context compaction — the session manages its own context window:

```js
await joinSession({
  infiniteSessions: {
    enabled: true,
    backgroundCompactionThreshold: 0.80, // start compacting at 80%
    bufferExhaustionThreshold: 0.95,      // block until done at 95%
  },
  onPermissionRequest: approveAll,
});
```

When enabled, sessions persist state to a workspace directory (`session.workspacePath`) with checkpoints, a plan file, and working files.

## BYOK: Bring Your Own Key

Point the session at any OpenAI-compatible API:

```js
await joinSession({
  provider: {
    type: "openai",      // or "azure" or "anthropic"
    baseUrl: "http://localhost:11434/v1", // Ollama, etc.
    apiKey: "optional-for-local",
    wireApi: "completions", // or "responses" for the newer API format
  },
  onPermissionRequest: approveAll,
});
```

## Scaffolding a New Extension

The fastest path from zero to working extension:

1. In a Copilot CLI session, the agent can call `extensions_manage({ operation: "scaffold", name: "my-ext" })` — this creates `.github/extensions/my-ext/extension.mjs` with a working skeleton.

2. For user-scoped extensions (available across all repos), add `location: "user"`.

3. Edit the generated file. Add your tools and hooks.

4. Reload with `extensions_reload({})` — the extension loads immediately, mid-session.

5. Verify with `extensions_manage({ operation: "list" })` and `extensions_manage({ operation: "inspect", name: "my-ext" })`.

The whole cycle — scaffold, edit, reload, test — happens in one session without losing context.

## What to Build

Given what the SDK *actually* offers:

**Security guardrails via `onPreToolUse`**: Intercept every tool call. Block dangerous shell commands. Deny writes to protected paths. Log everything. This is the highest-value extension pattern — it directly addresses the "AI agent ran amok" risk.

**Domain-specific tools via MCP**: Stand up an MCP server in whatever language your team uses. Expose internal APIs, database queries, monitoring dashboards as tools the agent can call.

**Custom agents for specialized workflows**: A `security-reviewer` agent, a `db-admin` agent, a `docs-writer` agent — each with a focused prompt and restricted tool set. Users switch with `/agent`.

**Structured decisions via elicitation**: When your extension needs explicit human confirmation — deployment targets, destructive operations, environment selection — listen for `elicitation.requested` events and present structured forms instead of relying on natural language.

**Prompt enrichment via `onUserPromptSubmitted`**: Automatically inject team context, coding standards, or relevant documentation into every prompt. The user types their question; your hook adds what they'd otherwise forget.

## The Real Extensibility Story

The Copilot CLI extension SDK is genuinely powerful. `joinSession()` with custom tools, six lifecycle hooks, MCP integration, custom agents, event-based elicitation, permission control, infinite sessions, BYOK support, and real in-session extension management covers serious ground.

What the SDK *doesn't* have is custom slash commands registered by extensions or a direct `session.ui.elicitation()` method call. The previous article was wrong to claim those. But what *is* here — tools, hooks, MCP, custom agents, structured elicitation events, and a full extension lifecycle with scaffold/reload/inspect — is more than enough to build meaningful governance, security, and productivity extensions.
